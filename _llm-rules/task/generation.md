# generate_taskコマンド専用ルール

## 🎯 基本原則

### 明示的実行のみ
**重要**: このコマンドは明示的な実行時のみ動作します。

- ✅ 許可される実行方法:
  ```
  mcp__ai-spec-driven-document__generate_task order="機能名を実装したい" target_repo="[リポジトリ名]"
  ```

- ❌ 禁止される自動実行:
  - 「Issueを作ってください」等の曖昧な自然言語
  - 「タスクを分解して」等の間接的な要求
  - コンテキストからの推測による実行

### コマンドの目的
`generate_task`は以下のプロセスをサポートします：
1. **要件分析**: 自然言語の要件を分析
2. **タスク分解**: 適切な粒度でタスク化
3. **Issue候補生成**: DRAFT形式でIssue候補を生成

## 📋 実行ポリシー

### 実行前の確認事項
1. **明示的なコマンド指定**: `generate_task`が明確に指定されているか
2. **必須パラメータ**: `order`と`target_repo`が提供されているか
3. **ユーザーの意図**: タスク分解が本当に必要か

### 自動実行の防止
- セッション中の暗黙的なタスク生成要求は無視
- 「Issue作成」の文脈でも`generate_task`は自動実行しない
- ユーザーが明示的にコマンドを指定した場合のみ実行

## 🔧 使用方法

### 基本的な使い方
```bash
# 機能開発のタスク分解
mcp__ai-spec-driven-document__generate_task \
  order="ユーザー認証機能を実装したい" \
  target_repo="[リポジトリ名]"

# 追加コンテキスト付き
mcp__ai-spec-driven-document__generate_task \
  order="管理画面のダッシュボードを作成" \
  target_repo="[リポジトリ名]" \
  context="React 19.1.0, TypeScript 5.8, Vite, Chakra UIを使用"
```

### パラメータ説明
- **order** (必須): 自然言語での要件・要望
- **target_repo** (必須): 対象リポジトリ
- **context** (任意): 追加の技術的コンテキスト
- **force_regenerate** (任意): キャッシュを無視して再生成
- **save_to_draft** (任意): DRAFTセクションに自動保存

## 📊 タスク分解のベストプラクティス

### 適切な粒度
- **1タスク = 1-3日の作業量**
- **独立性**: 他のタスクに依存しない単位
- **テスト可能**: 完了条件が明確
- **価値提供**: 具体的な価値を生む単位

### 3段階ワークフロー
1. **設計フェーズ**: 技術選定、アーキテクチャ設計
2. **実装フェーズ**: 実際のコーディング作業
3. **検証フェーズ**: テスト、品質確認

### タスクの構造化
```
要件（自然言語オーダー）
    ↓ 分析
技術要素の特定
    ↓ 分解
フェーズ別タスク
    ↓ 生成
Issue候補（DRAFT）
```

## ⚠️ 注意事項

### 実行時の制約
- コマンド名の正確な入力が必要（`generate_task`単数形）
- リポジトリ名は既定の値から選択
- 生成結果は候補であり、レビューが必要

### 他ツールとの連携
- 生成されたDRAFTは`sync_with_github`で実Issue化
- `check_duplicates`で重複確認を推奨
- `manage_github_projects`でProjects連携可能

## 🔄 ワークフロー

### 推奨される実行手順
1. **要件確認**: 明確な要件があるか確認
2. **コマンド実行**: `generate_task`を明示的に実行
3. **結果確認**: 生成されたタスク候補をレビュー
4. **DRAFT保存**: 必要に応じて`save_to_draft=true`
5. **Issue作成**: `sync_with_github`で実Issue化

### エラー時の対応
- パラメータ不足: 必須パラメータを確認
- リポジトリ名誤り: 正しいリポジトリ名を使用
- 生成失敗: `force_regenerate=true`で再試行
