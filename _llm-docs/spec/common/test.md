# シナリオテスト項目作成 指示書

## 指示書の目的

本指示書は、仕様書を基にシナリオテスト項目を効率的かつ網羅的に作成するための統一されたガイドラインです。LLMを活用してテスト項目を抽出し、プロジェクト横断で利用可能な高品質なテストドキュメントを作成することを目的とします。

## 作成手順

### Step 0: 対象プロジェクトの確認（必須）
**重要**: 対象プロジェクトが明確に指定されていない場合は、以下のエラーメッセージを表示して処理を中断すること。
```
エラー: 対象プロジェクトが指定されていません。
```

### Step 1: 参照仕様書の指定
1. **プロジェクト識別子**（例: fuji / kyoto）を明記する
2. **共通参照ディレクトリ**として `_llm-docs/spec/common` を必ず含める
3. **案件固有ディレクトリ**（例: `_llm-docs/spec/fuji`）をプロジェクトに応じて追加する
4. 参照するファイルとセクション範囲を箇条書きで列挙し、不足があれば「不足情報」として記録する

### Step 2: 対象仕様書の分析
1. 対象となる仕様書・設計書を特定する
2. 機能要件・画面設計・API仕様を整理する
3. テスト対象機能の範囲を明確にする

### Step 3: テストスイート設計
1. 機能単位でテストスイートを分割する
2. 各スイートのスコープを明確に定義する
3. 状態遷移・画面遷移フローを洗い出し、進行ステータスごとの分岐（例: 進捗状況、認証状態）を整理する

### Step 4: テスト項目抽出
1. 正常系・異常系の観点から項目を洗い出す
2. 必須確認観点（後述）に基づいて漏れなく抽出する
3. 実施手順と期待結果を具体的に記述する
4. 洗い出した状態分岐・画面遷移フローを必ずテスト項目に反映する（状態別活性制御、遷移元/遷移先、失敗時挙動を含める）

## 出力形式規則

### LLM入力テンプレート
```markdown
[プロジェクト名] シナリオテスト抽出依頼
- 対象プロジェクト: [プロジェクト識別子]
- 参照ディレクトリ:
  - _llm-docs/spec/common
  - _llm-docs/spec/[プロジェクト識別子]
- 参照ファイル:
  - [共通仕様ファイル名]
  - [案件仕様ファイル名]（セクション[開始]〜[終了]）
- 不足情報: なし（不足時は「不足情報: ...」と明記）
- 出力言語: 日本語
```

### 出力ファイル配置
- 完成したテスト項目は `_llm-docs/spec/[プロジェクト識別子]/scenario-test.md` に保存する
- 既存ファイルがある場合は上書きし、ない場合は新規作成する

### 全体構成
```markdown
# [プロジェクト名] シナリオテスト項目

## 目次
- [テストスイート名](#テストスイート名)
- [テストスイート名](#テストスイート名)
...

## [機能名] / [画面名]
[各テストスイートの詳細]
```

### テストスイートの必須構成要素

#### スコープ定義（必須）
```markdown
### スコープ
- **前提**: [何の機能のテストか]
- **画面レイアウト**: [どのページ/UIのテストか]
- **対象仕様書**: [ファイル名.md]
```

#### テスト項目表（必須）
正常系と異常系を分けて以下の表形式で記載：

| No | 合格 | 確認観点 | 期待結果 | 実施手順 | 備考 |
|----|------|----------|----------|----------|------|
| 1 | - [ ] | [テストの焦点] | [可観測で具体的な結果] | [番号付き手順] | [仕様ID等] |

※ 実施手順は表セル内で `1.` `2.` 形式を `<br>` 区切りで記載する。
※ テストデータや前提状態は備考欄に `データ:` `前提:` プレフィックスを付けて記述する。
※ 合格カラムは Markdown チェックボックス（`- [ ]`）のみを記載し、合格時のみ `- [x]` に更新する。

## 必須確認観点

### 正常系テスト
- **基本機能動作**: 主要機能が設計通りに動作する
- **画面表示**: UI要素が正しく表示される
- **データ処理**: 入力データが適切に処理される
- **遷移動作**: 画面遷移が正常に行われる
- **外部連携**: API・外部サービスとの連携が正常

### 異常系テスト
- **入力検証**: 不正な入力値での動作確認
- **エラーハンドリング**: エラー発生時の適切な処理
- **境界値**: 最大値・最小値での動作確認
- **ネットワーク異常**: 通信エラー時の動作
- **権限チェック**: 認証・認可の確認

### アクセシビリティ・ユーザビリティ
- **操作性**: 直感的な操作が可能
- **レスポンシブ**: 各デバイスでの表示確認
- **パフォーマンス**: 処理時間の妥当性

## テスト項目表の記述ルール

### 確認観点
- 何をテストするかを簡潔に表現
- 「〜を確認する」形式で記載
- 例：「ログイン認証の成功確認」「必須項目未入力時のエラー表示確認」

### 期待結果
- 可観測で具体的な結果を記載
- UI表示・状態変化・遷移・メッセージを具体的に
- 例：「ダッシュボード画面に遷移し、ユーザー名が表示される」

### 実施手順
- 誰が実行しても再現可能な粒度で番号付き
- 具体的な操作を順序立てて記載
- 例：「1. ログイン画面を開く 2. 正しいメールアドレスを入力 3. ログインボタンをクリック」

### 判定
- 合格カラムには Markdown チェックボックス（`- [ ]`）のみを記載し、合格時のみ `- [x]` に更新する

### 備考
- 関連する仕様ID、注意事項、制約条件を記載
- 必要に応じて関連画面IDや外部連携情報を追記

## 記述例

```markdown
## LINE認証 / ログイン画面

### スコープ
- **前提**: LINE認証機能、ログイン画面
- **画面レイアウト**: ログインフォーム、認証ボタン
- **対象仕様書**: auth-specification.md

### テスト一覧（正常系）
| No | 合格 | 確認観点 | 期待結果 | 実施手順 | 備考 |
|----|------|----------|----------|----------|------|
| 1 | - [ ] | LINE認証成功での画面遷移 | 認証成功後にメイン画面に遷移する | 1. ログイン画面を開く<br>2. LINE認証ボタンをタップ<br>3. LINE認証を完了する | REQ-AUTH-001 |
| 2 | - [ ] | ユーザー情報の取得・表示 | 認証後にニックネームが画面上部に表示される | 1. LINE認証を完了<br>2. メイン画面でユーザー情報を確認 | プロフィール表示要件 |

### テスト一覧（異常系）
| No | 合格 | 確認観点 | 期待結果 | 実施手順 | 備考 |
|----|------|----------|----------|----------|------|
| 3 | - [ ] | 認証キャンセル時の動作 | ログイン画面に戻り、エラーメッセージが表示される | 1. LINE認証ボタンをタップ<br>2. LINE認証画面でキャンセルを選択 | MSG-AUTH-CANCEL |
| 4 | - [ ] | ネットワークエラー時の処理 | 通信エラーメッセージが表示され、リトライボタンが利用可能 | 1. ネットワークを無効化<br>2. LINE認証を試行 | エラーハンドリング要件 |
```

## 品質確認チェックリスト

### 構成確認
- [ ] 各テストスイートに前提・画面レイアウト・対象仕様書が明記されている
- [ ] 正常系・異常系の両方が含まれている

### 内容確認
- [ ] 確認観点が具体的で理解しやすい
- [ ] 期待結果が「可観測かつ具体的」になっている
- [ ] 実施手順が「番号付き」で再現可能
- [ ] 判定列が「合格」とチェックボックス（- [ ]）で記載され、合格時のみチェックが入るようになっている

### 網羅性確認
- [ ] Step 0 でプロジェクト識別子と参照ディレクトリが指定されている
- [ ] 共通ディレクトリと案件固有ディレクトリの両方が入力テンプレートに含まれている
- [ ] 参照ファイルがセクション単位で列挙され、不足時は「不足情報」として記載されている
- [ ] LLMへの指示と出力テンプレートが一致している

### 一貫性確認
- [ ] 用語や表記にゆれがない
- [ ] テスト項目の粒度が統一されている
- [ ] 記述形式が統一されている

## 実施における注意点

### 効率的な作成のために
1. **段階的作成**: まずテストスイート構成を決めてから詳細を記述
2. **既存資産の活用**: 類似プロジェクトのテスト項目を参考にする
3. **レビューサイクル**: 作成後は必ず関係者によるレビューを実施

### 品質向上のために
1. **実際の利用シーンを想定**: ユーザーの実際の行動パターンを考慮
2. **エラーケースの充実**: 異常系テストを十分に検討
3. **継続的改善**: テスト実施結果を踏まえて項目を改善

この指示書に従うことで、統一された高品質なシナリオテスト項目を効率的に作成できます。

### 不足情報報告ルール
- 指定ファイルが存在しない場合は推測せず「不足情報」として列挙する
- 引用セクションが未記載の場合は TODO として明示し、再指示を促す
- 仕様書に記載のない事項については項目を作成せず、不明点として報告する
