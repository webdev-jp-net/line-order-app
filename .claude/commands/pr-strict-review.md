📝 PR厳格レビューモード - Claudeバイアス補正視点

## 使用方法
```
/pr-strict-review <PR番号>
例: /pr-strict-review 233
```

## 🎯 レビュー基本原則（Claudeの衝動バイアスを抑制）
- **仕様書＝唯一の真実** - 仕様書に記載のない実装は却下
- **効率化＝却下理由** - 勝手な最適化は品質リスク
- **一般化＝違反** - プロジェクト固有の要件を優先
- **省略＝不合格** - 完全性なくして品質なし
- **既存尊重** - 動いているコードを「改善」しない

## 🔍 テックリードレビュー観点

### 実装原則の遵守（implementation_principles.md準拠）

#### 既存コード影響評価
- [ ] 正常動作している既存コードを変更していないか
- [ ] 要件に直接関係ない部分の修正をしていないか
- [ ] 変更の影響範囲が明確に把握されているか
- [ ] 互換性が保たれているか

#### エラーハンドリング品質
- [ ] 想定エラーケースが網羅されているか
- [ ] エラーメッセージがユーザーフレンドリーか
- [ ] 適切なログ出力が実装されているか
- [ ] リトライ処理が必要な箇所に実装されているか

#### セキュリティ実装
- [ ] 入力値検証が全外部入力に実装されているか
- [ ] SQLインジェクション・XSS対策が実装されているか
- [ ] 機密情報が環境変数で管理されているか
- [ ] セキュリティヘッダーが適切に設定されているか

#### コード品質と統一性
- [ ] プロジェクトの命名規則に従っているか
- [ ] 既存コードパターンを踏襲しているか
- [ ] 不要なコメント・デバッグコードが削除されているか
- [ ] フォーマッター・リンターエラーが解消されているか

### タスク管理原則（core_rules.md準拠）

#### 仕様との整合性
- [ ] 実装が仕様書と完全に一致しているか
- [ ] 仕様変更があった場合、仕様書も更新されているか
- [ ] 憶測や推測による機能追加がないか
- [ ] 未記載事項は「仕様未定」と明記されているか

#### 重複実装の防止
- [ ] 類似機能の既存実装を確認したか
- [ ] 同一名称の関数やコンポーネントがないか
- [ ] 共通化できる処理を適切に配置しているか
- [ ] 重複するAPIエンドポイントがないか

#### 品質管理プロセス
- [ ] 各機能の動作確認が実施されているか
- [ ] エラー発生時の原因特定と対策が記録されているか
- [ ] デバッグログの確認と分析が行われているか
- [ ] 検証結果が適切に記録されているか

### 技術的判断の優先順位
1. **セキュリティ** - 脆弱性対策を最優先で確認
2. **パフォーマンス** - 処理速度とリソース効率を評価
3. **可読性** - コードの理解しやすさと保守性を確認

## 📊 レビュー出力フォーマット

### 必須出力項目
- **仕様差分マトリックス**
  - 仕様書記載事項 vs 実装内容の対照表
  - 差分がある場合は理由と対応方針

- **リスク評価**
  - セキュリティリスク: [High/Medium/Low]
  - パフォーマンスリスク: [High/Medium/Low]
  - 保守性リスク: [High/Medium/Low]

- **改修必須項目**
  - 緊急度High: セキュリティ・重大バグ
  - 緊急度Medium: 仕様不整合・品質問題
  - 緊急度Low: コード品質・命名規則

- **承認判定**
  - ✅ Approved: 全項目クリア
  - 🔄 Changes Requested: 改修必須
  - ❌ Rejected: 根本的な問題あり

## 🚨 即却下項目（テックリード権限）
- 仕様書にない機能の追加
- セキュリティ脆弱性の存在
- 既存機能への予期しない影響
- テストなしでの重要機能変更
- 環境変数以外での機密情報管理
- 技術スタックの無断変更

## 💡 レビュー実施手順
1. 仕様書と実装の完全照合
2. 上記チェックリストの全項目確認
3. リスク評価と改修項目の洗い出し
4. 仕様差分マトリックスの作成
5. 承認判定と具体的な改修指示

---
**注**: このレビューは品質保証の最終防衛線。妥協なき厳格な審査を実施すること。
